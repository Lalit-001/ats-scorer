# ATS Resume Scorer

AI-driven Applicant Tracking System. An admin creates a Job Description, candidates apply
with a PDF resume, and an asynchronous pipeline extracts, structures, and evaluates each
resume against the JD to produce a match score and recommendation.

## Stack

- **Frontend:** React + Vite + TypeScript
- **API:** Python + FastAPI (`api-py`, host `:4001`)
- **Worker:** Python + ARQ (`worker-py`, pipeline orchestration)
- **Parser:** Python + FastAPI + PyMuPDF (PDF extraction only)
- **LLM:** Gemini 2.5 Flash over raw HTTP / `httpx` — no SDK (rotating API key pool)
- **Data:** PostgreSQL (SQLAlchemy 2.0 + Alembic) + Redis
- **Local:** Docker Compose

> The backend was migrated from Node (Express · Sequelize · BullMQ) to Python. The original Node
> stack (`api` / `worker`) is retained behind a Docker Compose `legacy` profile and is **not**
> started by default — run it with `docker compose --profile legacy up`.

- [`docs/ARCHITECTURE.md`](docs/ARCHITECTURE.md) — system diagrams (topology, sequence, orchestration, key pool, data model)
- [`docs/superpowers/specs/2026-06-17-ats-resume-scorer-design.md`](docs/superpowers/specs/2026-06-17-ats-resume-scorer-design.md) — full design spec

## Prerequisites

- Docker + Docker Compose
- One or more **Gemini API keys** (free tier) from https://aistudio.google.com/apikey.
  Create the key under several Google accounts and list them comma-separated to multiply the
  free-tier quota — the worker rotates across them and fails over on rate limits.

## Run it

```bash
cp .env.example .env
#   edit .env:
#     GEMINI_API_KEYS=key1,key2,key3      <- your real keys
#     ADMIN_PASSWORD=something-you-pick
docker compose up --build     # Python stack: api-py, worker-py, parser, db, redis, mailpit, frontend
```

This brings up the **Python** backend by default. To run the legacy Node backend instead, use
`docker compose --profile legacy up` and point `VITE_API_BASE`/`MP_WEBHOOK_URL` back at
`:4000`/`@api`.

On boot the `api-py` service runs `alembic upgrade head` to apply pending migrations before
serving. Schema is owned entirely by migrations — there is no auto-create. **First run on an empty
database** needs the tables created once — see [Database migrations](#database-migrations).

Services and ports:

| URL | What |
|-----|------|
| http://localhost:5173/admin | Admin dashboard (log in with `ADMIN_PASSWORD`) |
| http://localhost:5173/apply/:slug | Candidate apply page (link shown after you create a JD) |
| http://localhost:4001/health | API health (Python `api-py`) |
| http://localhost:8000/health | Parser health |
| http://localhost:8025 | Mailpit — inbound-email testing UI |

## Demo flow

1. Open **http://localhost:5173/admin** and log in with `ADMIN_PASSWORD`.
2. **Create a job description.** Copy the generated apply link.
3. Open the apply link, fill in name + email, and upload a **PDF resume**.
4. Back on the job's applicants page, watch the row move
   `uploaded → processing → completed` (polled live). Open a row to see the parsed resume,
   detected certificates/links, match score, strengths/gaps, and recommendation.
5. If a stage breaks (e.g. Gemini quota), the row shows **failed** with the broken stage and a
   **Re-process** button.

## Database migrations

Schema changes go through **Alembic** (the SQLAlchemy migration tool) in
`backend-py/alembic/versions/`. `api-py` runs `alembic upgrade head` on boot. Run these from
`backend-py/` (or inside the container, e.g. `docker compose exec api-py alembic …`):

```bash
alembic upgrade head                          # apply pending migrations
alembic downgrade -1                          # roll back the most recent migration
alembic current                               # show the current revision
alembic history                               # list revisions
alembic revision --autogenerate -m "add_x"    # scaffold a migration by diffing the models
```

Review autogenerated files before applying; Postgres enum value additions need a manual
`op.execute("ALTER TYPE … ADD VALUE …")`. The DB connection comes from `DATABASE_URL` (converted to
an asyncpg DSN in `backend-py/app/config.py`). Alembic tracks its state in `alembic_version`,
separate from Node's `SequelizeMeta`.

**Fresh / empty database:** the current Alembic baseline *adopts* an already-existing schema (it was
created via `alembic stamp head`, so its `upgrade()` is empty). A brand-new DB therefore needs the
tables created once before the Python stack can use them — run the legacy Node migrations a single
time:

```bash
docker compose --profile legacy run --rm api npm run migrate
```

The legacy Sequelize migrations remain in `backend/migrations/` as the historical record.

## Troubleshooting

- **Everything `failed` at `evaluate`** → check `GEMINI_API_KEYS` are valid and not over quota
  (`docker compose logs worker-py`).
- **`failed` at `extract`** → the parser couldn't read the PDF; check `docker compose logs parser`.
- **Reset the database** → `docker compose down -v` (drops the Postgres, uploads, and Mailpit inbox
  volumes), then recreate the schema (see [Database migrations](#database-migrations)).
